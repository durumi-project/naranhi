## 0. 한 줄 요약

**Vercel 프로덕션의 모든 함수(`/api/classify`, `/api/ping` 포함)가 30초 timeout 으로 응답을 못 돌려준다.** 5줄짜리 ping 함수까지 막혀서 코드 자리 문제가 아니라 *Vercel Function 인프라 자체* 의심 단계. 로컬에서는 모든 게 정상 작동.

## 3. 현재 막힌 자리 — 무엇이 안 되는지

### 3-1. 증상

- 프로덕션 URL `https://naranhi-nu.vercel.app` 의 `/api/classify` 호출 → **30초 timeout**
- 같은 도메인의 `/api/ping` (5줄짜리 진단 함수) 호출 → **같이 timeout**
- 즉, *함수 안의 코드와 무관*하게 함수 자체가 응답을 못 돌려줌
- Vercel 로그 상 *Function 시작* 은 보이는데 *응답 반환* 자리에서 막힘 (정확한 로그는 사용자 직접 확인 가능)

### 3-2. 영향 범위

- 학생 입력 → 결과 화면 LLM 친화 응답 카드: **고정 폴백 안내만 노출** (`clientCall.js` 의 `CLIENT_STAGE.TIMEOUT` 분기)
- 유사 사례 매칭: **정적 키워드 기반 폴백만 동작** (LLM 매칭 없이도 사례 1~3건은 표시됨)
- 안전 분기 (가정폭력·자해): **정상 작동** (백엔드 호출 전 클라이언트 키워드 1단계에서 차단되는 자리)

### 3-3. 작동하는 것

- `npm run dev` (localhost:5173) — *모든 기능 정상*. Vite 가 `scripts/devApiPlugin.mjs` 로 `/api/classify` 를 직접 마운트
- `npm run llm:test` — 5 시나리오 중 **4/5 통과** (1 실패 자리는 세션 19 신규 통합 자리에서 발생, 세션 14 자리에서는 5/5)
- `npm run build` — 통과 (152ms, dist 826KB)
- Anthropic API 키 자체 — 정상 (로컬에서 호출 통과)

## 4. 시도해본 진단 — 세션 12~19 흐름

### 4-1. 세션 12 (2026-05-16) — 첫 배포 디버그

- `vercel.json` `functions` 블록 문법 충돌 → 제거 → 빌드 통과

- `runtime` 값 `'nodejs20.x'` 거부 → `'nodejs'` 로 수정 → **정상 배포 + 시나리오 A·B·C 통과**

### 4-2. 세션 13 — tool_use 도입 → timeout 시작

- 프로덕션 호출 16건 *전수 504 timeout* (Vercel Runtime Timeout Error: Task timed out after 300 seconds)

- 로컬 `npm run llm:test` 5/5 통과 → 푸시 후 프로덕션 회복 기대

### 4-4. 세션 15~19 (2026-05-17) — M3 통합 시도 + 인프라 의심

- 세션 15~17: M3 (Supabase pgvector + OpenAI 임베딩) 인프라 구축 완료
- 세션 16·17·19: 프로덕션 통합 시도 3회 → **모두 30초 timeout**
- 세션 19 후: HEAD `84e969f` (세션 14 작동 자리) 로 **force push 복귀**
- 그러나 *같은 timeout 재발* → 세션 14 자리에서는 작동했으나 오늘은 같은 자리에서도 timeout

- 결론: 코드 자리 문제 가능성 *현저히 낮음*, Vercel Function 인프라 자체 의심

## 5. 의심 가설 (우선순위 순)

### 가설 A — Vercel 프로젝트 설정 변경 (가능성 높음)

- 어제 (2026-05-16) 작동했던 자리가 오늘 (2026-05-17) 같은 코드로 timeout → *외부 변수 변경* 가능성
- 후보: Node 버전 (Vercel 프로젝트 설정의 Node Version), region, 함수 메모리 한도, 환경변수
- 확인: Vercel 대시보드 → Project Settings → Functions / Environment Variables / General

### 가설 B — 함수 콜드 스타트 + bundle 크기

- `api/classify.js` 가 `casesContext.generated.js` (사례 71건, 134KB 정적 import) 를 끌어들임
- Node runtime 콜드 스타트 + 큰 bundle 로드가 *Vercel 함수 초기화 한도* 를 초과할 가능성
- 단, `/api/ping` (5줄, 의존 0) 도 timeout 인 자리 자체가 이 가설의 *반증* — bundle 크기와 무관

### 가설 C — outbound 트래픽 인그레스 차단

- Anthropic API / Supabase 로 가는 outbound HTTPS 가 Vercel 인그레스에서 막힘
- 단, `/api/ping` 은 outbound 호출 0인데도 timeout → 이 가설도 *반증*

### 가설 D — Vercel 빌드/배포 자체 결함

- `vercel.json` 의 `functions` 블록이 *문법은 통과하나 실제 함수 등록이 안 됨*
- 또는 `scripts/buildCasesContext.mjs && vite build` 빌드 명령이 *함수 디렉토리에 부작용*
- 가능성 *가장 큼* (코드 자리는 모두 정상인 자리)

> **외부 분 봐주실 자리**: 가설 A, D 가 가장 유력. Vercel 대시보드 + 빌드 로그가 필요한 자리라 자율 모드 (push 금지) 에서는 막혔던 자리.

## 7. 도움 요청 — 무엇이 필요한지

### 7-1. 1순위 (5분 안에 가능하면)

- Vercel 대시보드 접속 권한 받아서 다음 확인:
  - **Functions 로그** — `/api/ping` 호출 시 *함수가 실제 실행되는지*, *어디서 행이 걸리는지*
  - **Project Settings → General** — Node Version, Region, Build & Development Settings
  - **Project Settings → Functions** — Function Memory, Maximum Duration 실제 적용값
  - **Environment Variables** — `ANTHROPIC_API_KEY` 가 *Production* 환경에 적용됐는지

### 7-2. 2순위 (필요 시)

- `vercel.json` 의 `functions` 블록과 `runtime` 자리 검토 — Vercel 공식 권장 패턴 부합 여부
- `package.json` 의 `engines.node` 와 Vercel 프로젝트 Node Version 일치 여부
- `vercel --debug` 또는 `vercel build --debug` 로 로컬 재현 시도

### 7-3. 3순위 (시간 여유 있으면)

- M3 (벡터 검색) 통합 자리의 timeout 자리도 같은 원인일지 검토 — 세션 16·17·19 의 변경 자리는 *force push 로 사라짐* 이라 git reflog 또는 사용자에게 별도 패치 요청 필요
